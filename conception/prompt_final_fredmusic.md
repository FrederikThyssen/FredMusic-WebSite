# Prompt final — POC complet site Fredmusic

J’aimerais que tu réalises un **POC complet pour le site Fredmusic**, un site vitrine premium pour un DJ spécialisé dans les **mariages, soirées privées, événements professionnels, animation DJ, sonorisation, mise en lumière et location de matériel**.

Je veux que tu crées toutes les pages en **mode statique**, sans base de données, sans API réelle, avec uniquement des **données mockées**.

Le but est d’avoir une base solide, propre et professionnelle, qui pourra ensuite être connectée à une base de données, une API, un système d’authentification, une interface d’administration et éventuellement un CMS.

Tu dois travailler en **mode expert développeur**, avec une vraie logique métier, une architecture claire, des composants réutilisables, une bonne UX, une excellente base SEO, un site responsive mobile, accessible et sécurisé selon les bonnes pratiques web modernes.

---

## Objectif du projet

Le site doit présenter Fredmusic comme un DJ événementiel **haut de gamme**, avec une identité visuelle premium.

L’univers principal du site est sombre, élégant et immersif :

- noir profond
- gris anthracite
- doré premium
- ambiance lumière, DJ, événementiel, matériel professionnel
- style moderne, luxueux, sérieux

Le site doit aussi comporter un univers plus clair et raffiné pour la partie mariage :

- blanc
- crème
- beige
- doré doux
- ambiance romantique, lumineuse et premium
- mise en avant de l’émotion et de l’accompagnement des mariés

L’idée est d’avoir deux ambiances visuelles complémentaires :

- **site principal sombre premium**
- **section mariage claire, élégante et romantique**

Inspire-toi fortement des visuels, maquettes et chartes graphiques fournis précédemment.

## Workflow utilisateur attendu

Le parcours utilisateur doit être fluide.

Exemple de parcours principal :

1. L’utilisateur arrive sur l’accueil.
2. Il découvre l’univers Fredmusic.
3. Il consulte les prestations.
4. Il clique sur une prestation pour voir les détails.
5. Il consulte la page mariage s’il prépare un mariage.
6. Il regarde la galerie.
7. Il demande un devis via le formulaire.
8. Il reçoit un message de confirmation mocké.

Parcours location :

1. L’utilisateur arrive sur la page Location de matériel.
2. Il découvre les catégories de matériel.
3. Il consulte une fiche matériel.
4. Il clique sur “Demander un devis pour ce matériel”.
5. Le formulaire de devis est prérempli ou orienté vers la location.
6. Il reçoit une confirmation mockée.

Parcours invité :

1. Un invité arrive sur une page événement via QR code.
2. Il consulte les infos de l’événement.
3. Il clique sur “Demander une musique”.
4. Il remplit le formulaire.
5. Sa demande apparaît dans l’état local.
6. L’admin peut voir la demande dans l’interface mockée.

Il ne doit pas y avoir de page morte.  
Toutes les pages doivent être reliées entre elles par la navigation, les boutons ou les CTA.

## Accessibilité obligatoire

Le site doit respecter les bonnes pratiques d’accessibilité web.

Objectif : obtenir un bon score sur Lighthouse et éviter les erreurs évidentes d’accessibilité.

À prévoir :

- HTML sémantique : `header`, `nav`, `main`, `section`, `article`, `footer`
- un seul `h1` par page
- hiérarchie correcte des titres `h2`, `h3`, etc.
- textes alternatifs pertinents sur les images
- labels visibles ou accessibles pour tous les champs de formulaire
- boutons avec textes compréhensibles
- liens explicites
- contraste suffisant entre texte et fond
- focus visible au clavier
- navigation clavier possible
- menu mobile utilisable au clavier
- états `aria-expanded`, `aria-controls` si menu burger
- `aria-label` quand nécessaire
- ne pas utiliser uniquement la couleur pour transmettre une information
- messages de confirmation accessibles
- messages d’erreur lisibles si validation simulée
- taille de texte confortable
- zones cliquables adaptées au tactile
- respect de `prefers-reduced-motion` pour limiter les animations si l’utilisateur le demande

Évite les animations trop agressives ou inutiles.

## Sécurité et bonnes pratiques web

Même si le POC est statique, il doit être codé avec les bonnes pratiques de sécurité.

À prévoir :

- ne jamais utiliser `dangerouslySetInnerHTML` sauf nécessité justifiée
- validation minimale côté front des formulaires
- nettoyage logique des entrées utilisateur simulées
- pas d’injection HTML dans les rendus
- aucun secret dans le code
- aucune clé API réelle
- pas de données sensibles en dur
- pas de dépendances inutiles
- types TypeScript stricts
- éviter les `any`
- prévoir une structure compatible avec une future API sécurisée
- prévoir une documentation expliquant les futures mesures :
  - validation serveur
  - rate limit
  - protection CSRF si nécessaire
  - protection XSS
  - authentification admin
  - rôles et permissions
  - sécurisation des uploads d’images
  - variables d’environnement
  - headers de sécurité

Prévoir dans la documentation une section “Sécurité future”.

## Ce qu’il ne faut pas faire

Ne fais pas un simple site vitrine figé sans logique.

Ne crée pas seulement une landing page.

Ne laisse pas de page non reliée.

Ne crée pas de fausse interface admin inutile : elle doit refléter une vraie logique métier future.

Ne mets pas tout dans un seul fichier.

N’écris pas trop de CSS classique.  
Utilise Tailwind au maximum.

Ne crée pas de vraie API ni de vraie base de données.

Ne mélange pas toutes les données directement dans les composants.

Ne néglige pas le mobile.

Ne néglige pas l’accessibilité.

Ne crée pas un site joli mais mauvais en performance.

Ne mets pas de contenu SEO artificiel ou bourré de mots-clés.

Ne mets aucune clé API réelle, aucun secret, aucune donnée sensible.
